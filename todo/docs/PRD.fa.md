# سند PRD — پروژه Todo Smart (نسخه وب، مبتنی بر Docker)

## 1. معرفی کلی پروژه
Todo Smart یک اپلیکیشن وب برای مدیریت هوشمند وظایف است که به کاربران اجازه می‌دهد کارهای خود را با جزئیات کامل ثبت، پیگیری، تحلیل و بهینه‌سازی کنند.  
این پروژه از ابتدا با معماری **چند-سرویسی (Database / Backend / Frontend)** و به‌صورت **Docker-based** طراحی می‌شود تا:
- لانچ اولیه سریع باشد
- توسعه مرحله‌ای و امن انجام شود
- انتقال به سرور واقعی فقط با تغییر آدرس‌ها و envها انجام شود

---

## 2. اهداف اصلی (Goals)
- ایجاد یک سیستم مدیریت وظایف چندکاربره (Multi-user)
- جداسازی کامل داده‌های هر کاربر
- پشتیبانی از احراز هویت (Login / Logout)
- امکان توسعه آینده (موبایل، نوتیفیکیشن، هوش مصنوعی)
- دپلوی ساده با Docker در محیط‌های مختلف

---

## 3. کاربران هدف (Target Users)
- کاربران فردی که می‌خواهند کارهای روزانه/هفتگی خود را مدیریت کنند
- کاربران فارسی‌زبان
- افرادی که به تحلیل عملکرد شخصی علاقه‌مندند

---

## 4. دامنه پروژه (Scope)

### در MVP (نسخه اولیه)
- ثبت‌نام و ورود کاربر
- ایجاد، ویرایش، حذف وظیفه
- تعیین وضعیت وظیفه (انجام‌شده، انجام‌نشده، تعویق)
- تعیین ددلاین
- مشاهده لیست وظایف
- اجرای کامل روی وب

### خارج از MVP (Later)
- نوتیفیکیشن
- تحلیل پیشرفته
- اپ موبایل
- اشتراک‌گذاری وظایف

---

## 5. ویژگی‌ها (Features)

### 5.1 مدیریت کاربران
- ثبت‌نام با ایمیل و رمز عبور
- ورود و خروج
- هر کاربر فقط به داده‌های خود دسترسی دارد

---

### 5.2 مدیریت وظایف (Task Management)
هر وظیفه شامل موارد زیر است:
- عنوان (Title)
- توضیح (Description)
- زمان انجام / ددلاین
- وضعیت:
  - انجام شده (DONE)
  - انجام نشده (NOT_DONE)
  - به تعویق افتاده (DELAYED)
- سطح اهمیت

عملیات قابل انجام:
- ایجاد
- ویرایش
- حذف
- تغییر وضعیت
- تغییر زمان (Reschedule)

---

### 5.3 تحلیل ساده (Analytics – فازهای بعدی)
- تعداد وظایف انجام‌شده / انجام‌نشده
- گزارش روزانه / هفتگی / ماهانه
- نمایش گرافیکی (بعداً)

---

### 5.4 رابط کاربری (UI / UX)
- نسخه وب (Web-first) و Responsive
- طراحی مبتنی بر قوانین تعریف‌شده در سند `UI_DESIGN_RULES_WEB.fa.md`
- رابط کاربری شاد، پاستلی، دوستانه و بدون استرس
- استفاده از Layout وب شامل Top Bar، Sidebar و Main Content
- تمام رنگ‌ها، فونت‌ها و فاصله‌ها Token-based بوده و Hardcode نمی‌شوند
- تمام متن‌ها فعلاً فارسی و راست‌چین هستند

---

## 6. الزامات غیرعملکردی (Non-Functional Requirements)

### 6.1 معماری
- سه سرویس مستقل:
  - Database Server
  - Backend API Server
  - Frontend Web Server
- ارتباط Frontend و Backend فقط از طریق API

### 6.2 Docker
- تمام سرویس‌ها باید Dockerized باشند
- اجرای پروژه فقط با docker compose
- عدم نیاز به نصب دستی دیتابیس یا بک‌اند روی سیستم

---

### 6.3 تنظیمات (Configuration)
- استفاده کامل از فایل‌های `.env`
- عدم Hardcode شدن:
  - URLها
  - Secretها
  - Tokenها
- وجود فایل `.env.example`

---

## 7. تست و تحویل مرحله‌ای
- توسعه پروژه به صورت **فازبندی‌شده**
- در پایان هر فاز:
  - اجرای تست
  - ارائه لاگ خروجی
  - توقف و انتظار برای تأیید کارفرما
- بدون تأیید، ورود به فاز بعدی مجاز نیست

---

## 8. فازهای توسعه (خلاصه)

### فاز 1 — Bootstrap
- ایجاد ساختار پروژه
- ساخت docker-compose و Dockerfileها
- تعریف env.example
- تست بالا آمدن سرویس‌های خالی

### فاز 2 — Database Schema
- راه‌اندازی PostgreSQL
- تعریف اسکیمای دیتابیس (User, Task, UserSettings)
- اجرای migration اولیه

### فاز 3 — Backend Core & Auth
- پیاده‌سازی Backend پایه
- احراز هویت (Register / Login / Logout)
- جداسازی داده‌های کاربران

### فاز 4 — Backend Task Management
- CRUD کامل وظایف
- تغییر وضعیت و زمان‌بندی مجدد وظایف

### فاز 5 — Frontend Auth & Layout
- پیاده‌سازی Layout وب (Top Bar + Sidebar)
- صفحات Login / Register

### فاز 6 — Frontend Task UI
- لیست وظایف
- ایجاد و ویرایش وظایف
- اتصال کامل به API

### فاز 7 — Settings
- تنظیمات کاربر (حداقلی)
- ذخیره تنظیمات در دیتابیس

### فاز 8 — Analytics (پس از MVP)
- داشبورد گزارش‌ها
- تحلیل ساده عملکرد

---

## 9. معیار موفقیت (Success Metrics)
- اجرای پروژه فقط با یک دستور docker
- امکان ثبت‌نام و ورود کاربر
- ایجاد و مشاهده وظیفه بدون خطا
- تطابق UI با سند `UI_DESIGN_RULES_WEB.fa.md`
- عدم نیاز به دخالت دستی توسعه‌دهنده در ساخت فایل‌ها

---

- در فاز اول فقط وب
- تمرکز بر سادگی و پایداری
- تصمیمات تکنولوژیک در سند TECH_STACK مشخص می‌شوند
